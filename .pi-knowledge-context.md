# DTR Project Knowledge Context for pi

## Project Overview

DTR is a CLI tool written in Rust that helps organize R data analysis projects as a directed acyclic graph (DAG) of analysis nodes. It is heavily inspired by git's backend architecture.

## Core Specifications

The canonical specification is in **`projectspec.txt`**. When implementing any feature:

1. **Always read `projectspec.txt` first** — it is the single source of truth.
2. If **any non-trivial ambiguity** arises about how something should work (behavior, edge cases, file formats, command flags, error handling, etc.), **re-read `projectspec.txt`** to resolve it.
3. If after re-reading `projectspec.txt` the answer is still unclear, **prompt the user for clarification** rather than guessing.

## Development Workflow

**Always follow this order for every feature or bugfix:**

1. **Write tests first** (unit tests / integration tests) that define the expected behavior.
2. **Write the implementation code** to make the tests pass.
3. **Iterate**: run tests, fix failures, refine.

This applies to all changes — new commands, new node types, edge cases, error handling, anything.

## Technical Expertise Requirements

This project demands expertise in three key areas:

- **Rust expertise**: The CLI is built in Rust. Use idiomatic Rust (e.g., `Result`/`Option`, iterators, pattern matching, `clap` for CLI parsing, `serde` for JSON, `sha1` for hashing). Strongly prefer the Rust standard library and well-established crates.
- **R expertise**: The code snippets stored in nodes are R/tidyverse functions (dplyr, ggplot2, glm, etc.) that chain via R's native `|>` pipe. Commands like `dtr run` and `dtr compose` execute or compose R code. Understand R's pipe semantics, tidyverse conventions, and RDS serialization.
- **Git backend knowledge**: The `.dtr` directory structure mirrors git internals: blob storage by SHA1 hash, node metadata as JSON (like git objects), markers (like git refs), a CWN file (like HEAD), and a cache directory. Model implementations after git's elegant simplicity.

## Coding Style & Paradigm

- **Strongly prefer simple, readable, and concise code.** Avoid over-engineering, premature abstractions, and unnecessary complexity.
- **Use functional coding paradigms.** Favor:
  - Pure functions with no side effects where possible.
  - Immutable data structures.
  - Function composition and chaining (iterators, `map`, `filter`, `fold`, etc.).
  - Pattern matching over conditional branches.
  - `Result`/`Option` for control flow over exceptions or panics.
- Use descriptive variable names but keep things short enough to be readable.
- Favor small, focused functions that do one thing well.

## Key Architectural Points

### .dtr Directory Structure
```
.dtr/
├── CWN              # Current working node ID (like git HEAD)
├── cwnout           # Text output of the last dtr run
├── next_id          # Incrementing u64 counter for node IDs
├── markers          # JSON object: marker name → node ID
├── packages         # JSON array of R package names (e.g. ["dplyr","ggplot2"])
├── blobs/           # Immutable R code, content-addressed by SHA1
├── nodes/           # Node metadata JSON, mutable in place
└── cache/           # Cached outputs, RDS format, content-addressed by SHA1
```

### Node Naming
Nodes use incrementing hex IDs with an "n" prefix: `n0`, `n1`, `n2`, ..., `n9`, `na`, `nb`, ...
The counter is stored in `.dtr/next_id` and incremented atomically per node creation.
Write operations (dtr write) create a new node with a new ID and update all references.

### Node JSON Format (Node struct)
```rust
pub struct Node {
    pub node_type: String,       // "input" | "process" | "chart" | "model" | "merge"
    pub parents: Vec<String>,    // parent node IDs
    pub parent_vars: Vec<String>, // variable names for each parent ("" = pipes via |>)
    pub children: Vec<String>,   // child node IDs
    pub blob: String,            // SHA1 hash of R code in blobs/
    pub cache: Option<String>,   // SHA1 hash of cached RDS output
}
```
- No `time` field — removed in favor of incrementing IDs (simpler, no chrono dependency).
- `parent_vars` is stored at merge creation time (marker name or `p_<id>` fallback).
- `#[serde(default)]` on `parent_vars` for backward compatibility with old nodes.

### Key Design Decisions

1. **Nodes mutated in place, blobs content-addressed**: Node files are mutable by ID. Blobs are immutable SHA1-addressed. Same R code → same blob hash → same file (deduplication).

2. **Cache format**: Unified RDS binary format. `dtr run`, `dtr run -r`, and `dtr cache` all produce RDS caches. `compose_node_cached` uses `readRDS()` to reload cached ancestors.

3. **compose_impl pattern**: The shared compose body takes a `fn` pointer for recursion. `compose_node` and `compose_node_cached` are thin wrappers — zero code duplication.
   ```rust
   fn compose_impl(dir, hash, recurse: fn(&Path, &str) -> Result<String, DtrError>) -> ...
   fn compose_node(dir, hash) { compose_impl(dir, hash, compose_node) }
   fn compose_node_cached(dir, hash) { /* cache check */ compose_impl(dir, hash, compose_node_cached) }
   ```

4. **Library imports**: `dtr add-lib <pkg>` adds to `.dtr/packages` (JSON array). `compose()`, `run()`, and `cache()` prepend `library(pkg)` calls to every generated R script. No need to put `library()` in node code.

5. **Marker resolution**: `resolve_ref()` tries marker name first, then treats the string as a node ID directly. Used by `dtr goto`, `dtr add merge`.

6. **Temp file naming**: `execute_r_script` uses PID + nanosecond timestamp to avoid race conditions between concurrent calls.

### DtrError Variants
```rust
pub enum DtrError {
    Io(std::io::Error),
    NoCurrentNode,
    InvalidNodeType(String),
    NodeNotFound(String),
    InvalidState(String),
    RExecError(String),    // Dedicated variant for R execution failures
    Json(serde_json::Error),
}
```

## Full Command Reference

| Command | Description | Library Function |
|---------|-------------|-----------------|
| `dtr init` | Initialize .dtr/ directory | `init()` |
| `dtr add input <code> [-m <name>]` | Add root input node | `add_input()` |
| `dtr add process <code>` | Add dplyr transform node | `add("process", ...)` |
| `dtr add chart <code>` | Add ggplot2 chart node | `add("chart", ...)` |
| `dtr add model <code>` | Add statistical model node | `add("model", ...)` |
| `dtr add merge <code> <parent...>` | Merge node (parents: IDs or markers) | `add_merge()` |
| `dtr add-marker <name>` | Nickname current node | `add_marker()` |
| `dtr add-lib <package>` | Register R package dependency | `add_lib()` |
| `dtr read` | Print current node's R code | `read_current()` |
| `dtr write <code>` | Replace current node's R code (invalidates caches) | `write_current()` |
| `dtr goto <target>` | Move to node (ID, marker, `..`, `.`) | `goto()` |
| `dtr delete [-r]` | Delete current node (recursive with -r, cleans caches) | `delete_current()` |
| `dtr compose` | Output runnable R script (with library imports) | `compose()` |
| `dtr run [-r]` | Execute current node (-r = force, ignores cache) | `run()` |
| `dtr cache` | Run and cache current node as RDS | `cache()` |

## Key Internal Functions

| Function | Purpose |
|----------|---------|
| `create_node()` | Creates node JSON, writes blob, gets next ID |
| `next_node_id()` | Reads, increments, writes `.dtr/next_id`, returns `n<hex>` |
| `compose_impl()` | Shared compose body, takes recurse fn pointer |
| `compose_node()` | Full ancestor composition |
| `compose_node_cached()` | Cache-aware: stops at nodes with valid cache |
| `resolve_ref()` | Resolves marker name → node ID |
| `derive_parent_var_from()` | Gets variable name for merge side-parent |
| `build_r_script()` | Prepends library() calls to composed script |
| `wrap_r_script()` | Wraps composed script with result capture + saveRDS |
| `execute_r_script()` | Shells out to Rscript, captures stdout |
| `clear_descendant_caches()` | Recursively clears node.cache downstream |
| `blob_write()` | Content-addressable R code storage |
| `hash_string()` / `hash_bytes()` | SHA1 hashing (Digest imported at module top) |
